# Day2 指示フロー v1.0

## 1. 目的

Day2 は高スコア獲得日ではなく、本番提出経路の成立確認日とする。
本日の主目的は、以下の確認を完了することである。

1. `predict()` を通る本番提出経路の成立確認
2. `inference_server.serve()` 経路の成立確認
3. silent fallback の構造的完全撤去の監査
4. Submission Error / timeout / crash / 0点 の切り分け
5. ローカル成立・本番成立・fail-fast 実証・silent fallback 監査を混同しない運用の徹底

---

## 2. 役割分担

### 2.1 ChatGPT

**役割:** 司令塔 / 統合責任者

担当範囲:
- Day2 の目的・評価軸・完了条件の固定
- 各エージェントへの指示文作成
- 指示順序管理
- 実行結果の一次整理
- 監査役への投入
- 監査結果の反映
- 最終統合報告書の作成
- 最終判定の実施

### 2.2 Claude Code

**役割:** コード生成主担当

担当範囲:
- Day2 対象コードの修正
- fail-fast / logging / fallback 撤去周辺の実装
- 必要な markdown / snippet / log 整理
- GitHub 正本更新

### 2.3 Codex

**役割:** 補助コード生成 / 下書き / 代替案作成

担当範囲:
- Claude Code の制限到達時の代替実装
- 差分案、下書き、補助コード生成
- ローカル commit ベースの補助作業

制約:
- GitHub 正本更新の主担当にはしない
- push 成功未確認の経路を本番主線にしない

### 2.4 Gemini

**役割:** 分析補助 / 観点追加

担当範囲:
- 貼り付けられたコード・ログ・提出結果の分析
- 独立した観点からの補助レビュー

制約:
- GitHub / Google Drive を読めない場合は読んだ前提で書かない
- 保存していない場合は保存したと書かない
- 統合しない
- 最終判定しない

### 2.5 Claude

**役割:** 監査専任

担当範囲:
- 証跡不足の指摘
- 推測混入の指摘
- 危険な断定の排除
- Day2 完了扱いの危険性評価
- 追加証跡要求
- 再判定

### 2.6 ユーザー

**役割:** Kaggle 実行・提出・エージェント起動担当

担当範囲:
- Kaggle 環境での実行
- Kaggle 提出
- 外部エージェントへの起動文送付
- GitHub 書き込み時の確認承認

制約:
- 各報告の統合や要約は担当しない
- 最終報告書の作成は担当しない

## 3. 正本・補助保存先

### 3.1 正本

GitHub リポジトリを正本とする。

### 3.2 補助保存先

Google Drive を補助保存先とする。
GitHub に直接書き込めない場合のみ、一時的に Google Drive へ保存してよい。
ただし、Google Drive 上のファイルは正本扱いせず、後で GitHub 正本へ反映する。

## 4. 基本フロー

**Step 1. ChatGPT が Day2 の目的・完了条件を固定**

最初に ChatGPT が、Day2 の目的、評価軸、完了条件を固定する。
これにより、各エージェントが異なるゴールで動くことを防ぐ。

**Step 2. ChatGPT が Claude Code に実装指示**

ChatGPT は、Claude Code 向けに Day2 実装指示を作成する。
ここでの対象は、主に以下である。
- `predict()` 周辺
- `_process_attempt`
- `_select_answer`
- `inference_server.serve()` 周辺
- 例外処理
- silent fallback 撤去
- logging 強化

**Step 3. Claude Code が実装・更新**

Claude Code は、Day2 対象コードを修正し、必要に応じて GitHub 正本へ反映する。
この段階で Day2 の本命コードを作る。

**Step 4. Claude Code が重い場合は Codex に切替**

Claude Code の制限到達、長文差分、重い処理等が発生した場合は、ChatGPT が Codex 向け補助指示を作成する。
Codex は、下書き・差分案・代替実装案の作成を担当する。

**Step 5. ユーザーが Kaggle で実行・提出**

コードが整ったら、ユーザーが Kaggle 環境でローカル検証および必要に応じて本番提出を行う。
ここは人手工程として残す。

**Step 6. 実行結果を回収**

回収対象は、主に以下とする。
- ローカルログ
- 提出結果メモ
- スコア情報
- エラー有無
- 必要なコード断片

**Step 7. ChatGPT が一次整理**

ChatGPT は回収した証跡をもとに、一次整理を行う。
ここでは以下を明確に分離する。
- 観測済み事実
- 未確認事項
- ローカル成立
- 本番成立
- fail-fast 実証有無
- silent fallback 監査論点

この段階では最終判定は行わない。

**Step 8. ChatGPT が Claude に監査指示**

一次整理結果をもとに、ChatGPT は Claude 向け監査指示を作成する。
Claude は、以下を中心に監査する。
- 証跡不足
- 推測混入
- 危険な断定
- Day2 完了扱いの危険性
- Pass / 未確認 / NoGo の妥当性

**Step 9. Claude が監査結果を返却**

Claude は監査結果を返し、必要な追加証跡や危険箇所を指摘する。

**Step 10. 必要なら Claude Code / Codex へ再指示**

監査で問題が出た場合、ChatGPT は必要に応じて Claude Code または Codex に再指示を行う。
これにより、
```
実装 → 実行 → 整理 → 監査 → 再修正
```
のループを回す。

**Step 11. ChatGPT が最終統合・判定**

最後に ChatGPT が以下を作成する。
- Day2 統合報告書
- 判定
- 判定根拠
- 次アクション

この時点で初めて、最終判定を出す。

---

## 5. 判定区分

### 5.1 Pass

以下を満たす場合に限る。
- 本番提出経路の成立に客観証跡がある
- silent fallback 撤去に反する証拠がない
- 重大未確認事項が閉じている

### 5.2 未確認 / 要確認

以下の場合は Pass にしない。
- ローカル成立のみ確認されている
- 本番成立を user self-report のみで支えている
- fail-fast 実証が不足している
- 本番ログや本番証跡が不足している

### 5.3 NoGo

以下の場合は NoGo とする。
- silent fallback 疑いが残る
- 本番経路が未成立
- 重大異常系の扱いが不明
- 監査上の重大欠陥が残存する

---

## 6. 運用上の禁止事項

1. ローカル成立と本番成立を混同しない
2. user self-report を objective evidence 扱いしない
3. 推測を事実として書かない
4. 他エージェントの報告を勝手に統合しない
5. 保存していないものを保存したと書かない
6. アクセスしていない repo / Drive / file を読んだ前提で書かない
7. ユーザーに統合作業をさせない

---

## 7. 運用上の補足

- GitHub が正本、Google Drive は補助保存先とする
- Claude Code は GitHub 正本更新の主担当とする
- Codex は補助コード生成担当とする
- Gemini は分析補助に限定する
- Claude は監査専任とする
- ChatGPT は司令塔・統合・判定担当とする
- ユーザーは Kaggle 実行・提出・起動担当に限定する
